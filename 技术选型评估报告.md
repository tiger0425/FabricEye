# FabricEye 技术选型评估报告

**版本**: v1.0  
**日期**: 2026年2月28日  
**项目**: FabricEye - AI面料验布系统  

---

## 一、执行摘要

FabricEye 项目面向小微企业，提供基于多模态大模型的AI面料验布解决方案。本报告针对以下技术领域进行全面评估：

| 技术领域 | 推荐方案 | 风险评估 |
|---------|---------|---------|
| 多模态LLM | **云端**: Qwen3-VL → Kimi k2.5<br>**本地**: InternVL 3.5 | 低 |
| 工业相机采集 | 海康MVS SDK + 10GigE + H.265 | 低 |
| 视频处理架构 | Python + SharedMemory + FFmpeg | 中 |
| 本地部署 | vLLM + InternVL 3.5-8B/14B | 中 |
| ERP对接 | RESTful API + 适配器模式 | 低 |

**核心建议**: 采用"云端验证 + 边缘部署"的双轨策略，优先使用 Qwen3-VL 进行MVP验证，后期迁移至 InternVL 3.5 本地部署。

### 2.1 候选模型对比

| 评估维度 | Kimi k2.5 | Qwen3-VL | InternVL 3.5 |
|---------|-----------|----------|--------------|
| **视觉理解** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **缺陷检测** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **颜色识别** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **视频处理** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **推理速度** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **API成本** | 中 ($0.6/M) | **极低** ($0.2/M) | N/A |
| **本地部署** | ⭐⭐ (1T参数) | ⭐⭐⭐ (235B) | ⭐⭐⭐⭐⭐ (1B-30B) |

### 2.2 模型深度分析

#### Kimi k2.5 (Moonshot AI)
**优势**:
- **Agent Swarm**: 支持100个子Agent并行调度，可分别处理色差、密度、破洞等专项检测
- **Thinking Mode**: 深度推理模式，能排除光线干扰，降低误报率
- 256K上下文，适合长视频分析

**劣势**:
- 1T MoE参数，API延迟较高
- 本地部署成本极高（需H200集群）
- 价格相对较高

**适用场景**: 复杂质检报告生成、工艺改进建议

#### Qwen3-VL (Alibaba Cloud)
**优势**:
- **工业视觉Benchmark领先**: MLPerf v6.0测试中表现优异
- **极致性价比**: API价格仅为同类1/3
- Flash版本推理速度极快
- 对细微纹理和空间关系感知强

**劣势**:
- 旗舰版(235B)本地部署需8xA100
- 开源生态相对较新

**适用场景**: MVP阶段云端快速验证、高频视频流检测

#### InternVL 3.5 (OpenGVLab)
**优势**:
- **本地化部署首选**: 1B-30B多尺寸可选
- **消费级硬件友好**: 8B版本可在RTX 4090运行
- **Token压缩技术**: Visual Resolution Router压缩50% Token
- 完全开源，可自由微调

**劣势**:
- 32K上下文限制（超长视频需分段）
- 需要自行部署维护

**适用场景**: 工厂边缘端生产部署

### 2.3 推荐实施路线

```
阶段一 (MVP验证): Qwen3-VL-Plus API
  ↓ 成本低，开发快，验证产品方向
阶段二 (深度赋能): Kimi k2.5 (Thinking Mode)
  ↓ 复杂报告生成，工艺分析
阶段三 (生产部署): InternVL 3.5-8B/14B + vLLM
  ↓ 本地部署，数据安全，长期成本低
```

## 三、工业相机视频采集方案

### 3.1 硬件接口选型

| 分辨率/帧率 | 推荐接口 | 带宽需求 | 备注 |
|------------|---------|---------|------|
| 4K@30fps | USB 3.0 / GigE | 3-6 Gbps | 基础配置 |
| 4K@60fps | USB 3.0 / 10GigE | 6-12 Gbps | 推荐配置 |
| 8K@30fps | 10GigE / CoaXPress | 20-40 Gbps | 高端配置 |
| 8K@60fps | CoaXPress | 40+ Gbps | 专业级 |

**FabricEye推荐**: 4K@60fps + 10GigE 接口，兼顾成本与性能

### 3.2 SDK选择

| SDK方案 | 性能 | 功能完整性 | 开发难度 | 推荐度 |
|--------|------|-----------|---------|-------|
| **MVS SDK** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | **强烈推荐** |
| GigE Vision标准 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 跨品牌场景 |
| DirectShow | ⭐⭐ | ⭐⭐ | ⭐⭐ | 不推荐 |

**MVS SDK核心优势**:
- 海康原生优化，多线程回调采集
- 完整Python封装（MvImport）
- 支持工业相机特有参数（触发、增益、曝光）

### 3.3 视频编码方案

**推荐编码**: **H.265 (HEVC)**
- 相比H.264节省40-50%存储空间
- 2026年主流GPU均支持硬编
- 使用PyAV调用`hevc_nvenc`实现GPU加速

**参数建议**:
- 帧率: 30-60 fps（根据布速调整）
- 码率: 4K@20-50Mbps, 8K@80-150Mbps (VBR)
- 像素格式: Bayer采集 → GPU转RGB

### 3.4 双路并行架构

**推荐方案**: **Python Multiprocessing + SharedMemory**

```
┌─────────────────────────────────────────────────────┐
│                   主进程 (采集)                       │
│  ┌──────────────┐    ┌──────────────────────┐       │
│  │ MVS SDK      │───→│ SharedMemory Buffer  │       │
│  │ 4K/8K采集    │    │ (零拷贝共享)          │       │
│  └──────────────┘    └──────────────────────┘       │
└─────────────────────────────────────────────────────┘
                          │
          ┌───────────────┴───────────────┐
          ▼                               ▼
┌─────────────────────┐      ┌─────────────────────┐
│ 存储进程             │      │ AI识别进程          │
│ ┌─────────────────┐ │      │ ┌─────────────────┐ │
│ │ PyAV + H.265    │ │      │ │ 预处理/缩放      │ │
│ │ GPU硬件编码      │ │      │ │ ↓               │ │
│ │ ↓               │ │      │ │ MLLM推理         │ │
│ │ MKV文件存储      │ │      │ │ ↓               │ │
│ └─────────────────┘ │      │ │ 结果输出         │ │
└─────────────────────┘      └─────────────────────┘
```

**关键设计要点**:
- 使用`multiprocessing.shared_memory`实现零拷贝
- 必须避开Python GIL，使用多进程而非多线程
- GPU预处理：使用`cv2.cuda`或NPP库

### 3.5 存储策略

| 格式 | 优点 | 缺点 | 推荐场景 |
|-----|------|------|---------|
| **MKV** | 容错性强，断电保护 | 兼容性略差 | **强烈推荐** |
| MP4 | 兼容性好 | 断电损坏风险 | 归档视频 |
| 原始帧 | 无损 | 存储开销极大 | 缺陷分析片段 |

**存储架构**:
```
本地NVMe SSD (热存储)
  ├── 原始4K/8K视频 (7天)
  └── AI检测结果
      
对象存储 (冷存储)
  ├── 压缩存档 (30天)
  └── 仅保留缺陷片段 (长期)
```

**分级保留策略**:
- 有缺陷视频: 保留原始4K/8K，长期存储
- 无缺陷视频: 7天后降采样为1080P或删除

## 四、视频AI检测架构

### 4.1 视频帧提取策略

**推荐方案**: **自适应抽帧 + 关键帧检测**

| 策略 | 实现方式 | 适用场景 |
|-----|---------|---------|
| 固定间隔 | 每N帧提取1帧 | 匀速运动的布匹 |
| 自适应抽帧 | 基于运动检测动态调整 | 变速/启停场景 |
| 关键帧提取 | 场景变化检测 | 快速定位缺陷段 |

**FabricEye建议**:
- 基础抽帧: 1-2秒/帧（根据布速）
- 运动分析: 使用光流法检测异常运动
- 时间窗口: 3-5秒为一个检测单元

### 4.2 LLM调用策略

**推荐架构**: **轻量筛选 + MLLM深度分析**

```
视频帧 → [传统CV预筛选] → [MLLM深度检测] → 结果聚合
         (YOLO/SSD)        (Qwen3-VL/InternVL)
          快速排除          精确分析缺陷类型
          正常区域          和原因
```

**Prompt工程建议**:
```python
system_prompt = """你是一位专业的面料质检专家。请仔细分析输入的面料图像，检测以下问题：
1. 表面缺陷: 破洞、抽丝、划痕、污渍
2. 色差问题: 色花、阴阳色、批次差异
3. 纹理异常: 密度不均、织法错误

请以JSON格式输出检测结果：
{
  "defects": [
    {
      "type": "缺陷类型",
      "location": "位置描述",
      "severity": "严重等级(1-5)",
      "confidence": 置信度
    }
  ],
  "overall_score": "综合评分(0-100)",

```

**Visual RAG增强**:
- 构建标准面料样本库
- 检测时传入参考样本进行对比
- 提升颜色识别和细微缺陷检测精度

### 4.3 后端架构设计

**推荐方案**: **Celery + Redis + FastAPI**

```
┌─────────────────────────────────────────────────────────┐
│                    FastAPI Web服务                       │
│  ┌────────────┐  ┌────────────┐  ┌──────────────────┐   │
│  │ 任务提交API │  │ 进度查询API │  │ 报告下载API      │   │
│  └────────────┘  └────────────┘  └──────────────────┘   │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                  Celery 任务队列                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │ 视频解码    │  │ AI检测任务   │  │ 报告生成    │     │
│  │ Worker-1    │  │ Worker-2..N │  │ Worker-N    │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
│                                                         │
│  消息代理: Redis/RabbitMQ                                │
│  结果存储: PostgreSQL + MinIO (文件)                     │
└─────────────────────────────────────────────────────────┘
```

**任务状态流转**:
```
PENDING → DECODING → ANALYZING → AGGREGATING → REPORTING → COMPLETED
             ↓           ↓            ↓            ↓
           FAILED/RETRY (失败可重试3次)
```

### 4.4 报告生成方案

| 格式 | 技术方案 | 优点 | 缺点 |
|-----|---------|------|------|
| **PDF** | ReportLab / WeasyPrint | 版式固定，打印友好 | 文件较大 |
| **HTML** | Jinja2 + 前端渲染 | 交互性强，可嵌入视频 | 依赖浏览器 |
| **Excel** | openpyxl | 数据可编辑 | 不够直观 |

**FabricEye推荐**: 
- 主要报告: PDF（含缺陷截图和评分）
- 详细分析: HTML交互式报告（可在视频中定位缺陷）

### 4.5 可视化标注

**时间戳定位**:
- 缺陷坐标: 视频帧号 + 像素坐标(x, y, w, h)
- 播放跳转: 点击缺陷列表跳转到对应时间点
- 热图叠加: 在视频上叠加缺陷密度热图

## 五、本地部署方案

### 5.1 部署框架对比

| 框架 | 适用场景 | 性能 | 易用性 | 推荐度 |
|-----|---------|------|-------|-------|
| **vLLM** | 生产环境高并发 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | **强烈推荐** |
| **Ollama** | 个人开发测试 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 开发阶段 |
| SGLang | 高性能推理 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 可选 |
| LMDeploy | 国产替代 | ⭐⭐⭐⭐ | ⭐⭐⭐ | 备选 |

**vLLM核心优势**:
- PagedAttention技术，95%+显存利用率
- 连续批处理(Continuous Batching)，高并发场景吞吐量高
- 支持张量并行和流水线并行

### 5.2 硬件配置建议

| 模型 | GPU配置 | 显存需求 | 预估性能 |
|-----|---------|---------|---------|
| InternVL 3.5-8B | RTX 4090 (24G) | 16-20GB | 10-20 fps |
| InternVL 3.5-14B | 2x RTX 4090 | 28-32GB | 8-15 fps |
| InternVL 3.5-8B (INT4) | RTX 3090 (24G) | 10-12GB | 15-25 fps |
| InternVL 3.5-8B | A100 40G | 16GB | 20-30 fps |

**FabricEye推荐配置**:
- **MVP阶段**: 单卡 RTX 4090 + InternVL 3.5-8B
- **生产阶段**: 2-4x RTX 5090 (32G) 或 A100 40G

### 5.3 部署命令示例

```bash
# 使用vLLM部署InternVL 3.5
vllm serve OpenGVLab/InternVL3_5-8B \
  --dtype bfloat16 \
  --tensor-parallel-size 1 \
  --max-model-len 8192 \
  --gpu-memory-utilization 0.85

# 4-bit量化部署（降低显存）
vllm serve OpenGVLab/InternVL3_5-8B \
  --quantization awq \
  --dtype auto \
  --max-model-len 4096
```

## 七、技术风险评估

### 7.1 高风险项

| 风险 | 影响 | 缓解措施 |
|-----|------|---------|
| 4K/8K视频带宽不足 | 丢帧，漏检 | 使用10GigE，启用巨型帧 |
| GPU显存不足 | 推理失败 | 使用INT4量化，降低batch size |
| MLLM误报率高 | 用户体验差 | 结合传统CV预筛选，设置置信度阈值 |

### 7.2 中风险项

| 风险 | 影响 | 缓解措施 |
|-----|------|---------|
| ERP接口变更 | 同步失败 | 适配器模式隔离变化 |
| 存储空间不足 | 数据丢失 | 分级存储策略，自动清理 |
| 模型推理延迟 | 实时性差 | 异步处理，降低用户期望 |

### 7.3 技术债务预警

1. **初期选型锁定**: 云端API切换成本高，需尽早设计抽象层
2. **数据格式演进**: 预留版本号字段，支持向后兼容
3. **硬件依赖**: 记录GPU/CUDA版本，便于迁移

## 十、结论与建议

### 10.1 核心技术选型结论

| 领域 | 最终推荐 | 理由 |
|-----|---------|------|
| 云端LLM | **Qwen3-VL** | 性价比最高，工业视觉能力强 |
| 本地LLM | **InternVL 3.5-8B** | 消费级硬件可运行，开源可微调 |
| 相机SDK | **MVS SDK** | 性能最优，功能完整 |
| 视频编码 | **H.265 + PyAV** | 压缩率高，GPU加速 |
| 并行架构 | **SharedMemory** | 零拷贝，低延迟 |
| 任务队列 | **Celery + Redis** | 成熟稳定，社区活跃 |
| ERP对接 | **适配器模式** | 扩展性强，维护简单 |
